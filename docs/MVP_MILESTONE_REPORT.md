# 校园社团管理系统 (CCMS) 阶段性里程碑报告 (MVP 阶段)

## 1. 项目概述与 MVP 达成进度
作为《软件工程导论》课程的阶段性成果，我们在 MVP（最小可行性产品）阶段，成功跑通了一个具备**多租户隔离 (Multi-Tenant)** 的核心系统雏形。

**当前达成节点：**
- **工程架构**：落实了前后端解耦的 Monorepo 架构（React + Express / MariaDB）。
- **权限与沙盒**：实现了基于 JWT 的身份验证防线，并落地了多租户拦截中间件 (Tenant Guard)，确保不同学校/组织的数据互不渗透。
- **核心业务验证**：打通了从“用户注册绑定租户 -> 管理员建群、添加成员 -> 提交活动表单 -> 平台多态审批”的完整数据闭环。

## 2. 核心架构踩坑与教训
在从零开始手搓及重构的过程中，我们遇到了不少设计层面的痛点，并总结出以下工程经验：

### 2.1 状态机演进必须依托专职接口
**问题**：最初在处理“活动审批”时，我们尝试让前端调用通用的单一更新接口（如 `PATCH /activities/:id`），试图直接把状态篡改为“已批准”。但在具有关联生命周期影响的企业级软件中，这种做法不仅无法触发相关的逻辑记录（例如独立的 `Approval` 审批流转单），还会被各类后端字面量拦截器强行阻断。
**改进**：剥离单一数据的 CRUD 修改与复杂的“状态迁移操作”。针对流转行为，必须设立独立的 Action 动作节点（如 `POST /submit-approval` 和 `POST /decision`），通过事务机制统一处理数据落表和状态推进。
> 💡 **我们的巧思实例：** 针对状态转移，我们弃用了死板的字段篡改。当老师点击通过时，其实是调用的 `POST /approvals/:id/decision`，载荷中只允许传 `decision: "approve" | "reject"`。这迫使系统必须经历一整套验证：查验此单不仅确实处于 `pending`，还要核查操作者身份是否匹配，随后才会在数据库级的事务 (Transaction) 里同时生成审批日志并更改活动状态。这种“专事专口”的设计让数据无死角可寻。

### 2.2 防腐层不可盲信内部请求方
**问题**：在开发“邀请拉人功能”时，初版 API 逻辑粗暴地读取了请求 Header 带来的识别令牌 (`req.user.id`) 并执行插入操作。结果当具有至高权限的系统管理员，试图把其他普通用户加入社团时，系统死板地识别成了“管理员正在疯狂拉自己加群”，最终因键冲突抛出 409 报错。
**改进**：凡是包含代为管理的越权代理逻辑模型，API 设计均须采用“客体优先拦截，主体设为降级”的过滤准则。后端接口必须优先辨识和提取投递目标的特征身份（`target_user_id`），而不是盲信系统中间件注入的请求发起者身份。
> 💡 **我们的巧思实例：** 在现在的 `clubController.js` 中，我们仅用一行极简的三元表达式 `const targetUserId = req.body.user_id ? Number(req.body.user_id) : req.user.id;` 就巧妙融合了“学生主动入社”和“管理员后台批量拉入”两套逻辑。不传参数就是自己加入，传了目标 ID 就是代办。一个接口搞定两种场景，还避免了权限隔离错乱的核心痛点。

## 3. 前沿范式：AI 大模型时代的工程体验
本次项目的重要特征是深度融合了多种人工智能辅助编程工具。这不仅改变了编码的方式，也让我们对现代软件工程生态有了全新的视角：

### 3.1 意图驱动开发的边界 (Vibe Coding)
随着大语言模型的加持，我们具象体验了何为意图驱动开发 (Vibe Coding)。开发者不必再执念于某种冷僻语法糖的具体敲击，可将重心转为“架构与逻辑约束”。通过梳理安全的业务流程，将系统的安全底线和交互需求拆解为结构化提示词 (Prompt)，再交由 AI 代理完成高速代码编织。我们意识到，在代码高产的今天，软件工程师勾画大局和制止漏洞蔓延的能力越发稀缺。

### 3.2 环境隔离与护栏构建 (Harness Engineering)
AI 代理极高的生产力有时也会伴随“代码幻觉”和逻辑发散，因此，刻意引入代码护栏变得尤为重要：
- 在外部数据进入核心业务逻辑层 (Controller) 前，我们大量铺设了厚度的拦截网，如强制多租户验证中间件与 Express Validator 校验器。不论代码的参数调用多么失控，一经查验身份越界则立刻阻断，绝不允许脏数据去试探核心数据库。
- 采用隔离预发沙盒思想（Harness），在推翻原有审批流并交由 AI 补写前，均在准备充足“吊舱”环境试验，保留安全熔断后一键重置的能力。
> 💡 **我们的巧思实例：** 在系统架构里加入的特殊中间件 `tenantGuard`！当前端发来请求获取“社团 A 表单”时，这层护栏会抓取你 Header 里的 `X-Tenant-Code` 参数。它是守门员，如果在 `TenantMembership` 白名单表里没查到你是这所学校的注册人员，连进入 Controller 读取任何业务数据的机会都不会给，直接暴力断开返回 `403 Forbidden`。把复杂的数据隔离判定前置到了核心链路门口，真正做到了防呆防错。

### 3.3 警惕自动化测试幻境：环境偏差与除错指引
在应用诸多时下的 AI Coding CLI 等智能体工具时，我们总结了一个极为隐蔽的通病：**AI 工具内嵌的测试环境通常无法完全复现真实的浏览器复杂交互。**
- **局限性溯源**：机器自测时，往往极度偏好执行底层的终端测试（例如通过发一个 `curl` 指令，返回了 `200 OK` 状态码即向主机宣告完工）。但进入实际页面的 Web 环境，面对诸如 React 组件的生命周期挂载差异、跨域复杂通信以及字段微小的命名错位时，整个页面依然会产生白屏或崩溃。
- **人工干预闭环**：光靠给 AI 模糊施加一句“页面好像有个模块卡住了，帮我修一下”是做无用功甚至导致过度修改的。我们必须要人工发挥开发者应有的素质：定位捕获真实的 Network 异常响应包、精准反馈 Console 面板内的错误追溯以及关键 Header 数据异象，在精确复现的提示语境下，才能指引其重新锁定正确的代码区块实施精确手术修复。
> 💡 **我们的巧思实例：** 在修复“审批不触发动作”时，AI 最初使用 `curl` 能成功向后端提交审批动作。但在实际前端交互中，UI 画面的下拉选单 `<select>` 抛来的状态载荷是 `"status": "approved"`，这被后端的严格校验器毫不留情打回（因为正确的键是 `"decision": "approve"`）。最后，反而是我们在云端驱动了一个极具拟真的端到端 (E2E) 模拟代理浏览器亲自点击鼠标验证，才让 AI 从 UI 返回拦截截图中觉察到了这种基于渲染流的隐藏微小漏洞。不能尽信后端跑通，必须端到端实操为王。

### 3.4 极速重构背景下的 Git 协同规范
AI 外挂机器极高的并发代码写出，逼迫我们重塑了版本控制理念：
- 有时单次对话往往裹挟着几百行前后端跨域的同步更新。为了防止由于牵一发而动全身所带来的毁灭级 Merge Conflict（代码冲突风暴），开发中我们必须铁腕推行 **原子级别提交 (Atomic Commits)**，做到改稳一个小切片立即存档封闭。
- Monorepo 的结构设计在此显得珍贵，我们在物理工程层面上将前端目录与后端微服务入口硬解耦，当大批量自动化脚本生成时不至于引起项目级别的关联依赖塌陷。

### 3.5 多节点智能体协同：以文档为载体的 Agent 知识网络
本项目并非单打独斗，而是多人编队的联合作战。值得一提的是，在这个过程中，每个组员都在调动属于自己的专属 AI Agent，这引发了一个新的挑战：如何让这些独立的机器助手统一思想？
- **沉淀文档作为 Agent 的中央记忆**：我们立下规矩，在每次重大代码提交或踩坑后，必须立刻以人类语言将故障点及修复规范写入根目录的 `experience.md` 以及配套的 `docs` 开发文档中。
- **让机器互相“跨时空交流”**：这些被严格维护的 Markdown 文档，本质上构成了全部 Agent 共享的知识图谱连接网络。当组员 A 的 Agent 踩了“租户鉴权覆盖”的坑并汇编成文后，组员 B 的 Agent 在进行后续功能拓展时，会主动读取该经验库。通过这种机制，我们惊喜地发现：通过代码库内的文档介质，我们不仅仅实现了人的协作，更成功实现了**不同 Agent 之间的跨时空经验继承与系统级学习**。
> 💡 **我们的巧思实例：** 针对“前端封装丢失头部信息”的大案，我们在 `experience.md` 里手写了一条死命令：*“必须先拆分 options 再合并 defaultHeaders，严禁对 options.headers 执行简单解构覆盖”*。自那以后，无论是谁的 AI Agent，但凡只要接到“增加新前端 API 路由”的代码指令，它读完这则规则后生成的 `apiRequest` 函数代码便会天然免疫此类覆盖 Bug。文档已经不再是写给人看的交接指南，而是直接成了**驯化和互联所有分布式智能体的“大模型配置文件”**！

---
**阶段总结**：
在 MVP 的磨合中，不单是编写了一套可用的社团模块，其真正的意义在于让我们明白：当敲代码的壁垒因 AI 逐渐消融，身为系统工程师的最强心智模型应落在——精准把控软件的骨架架构、用严密隔离网络托底，以及探索出一套让机器与人类开发心智真正对齐的进阶范式。
