## 2. 项目目标与范围（MVP）

面向校园社团场景，MVP覆盖四条主线能力：

1. 活动发布与管理
2. 成员招募与社团成员管理
3. 活动审批流（提交-审核-通过/驳回）
4. 财务公示（收支录入、公开展示）

角色采用三层权限：

- 学生（Student）
- 社团管理员（Club Admin）
- 系统管理员（System Admin）

## 3. 技术方案（与教学/软件工程实践匹配）

### 3.1 前端

- React（Create React App）
- React Router（路由）
- Axios（接口通信）
- Ant Design 或 Material UI（组件库）

### 3.2 后端

### 3.3 数据层

- MySQL 8.x

### 3.4 工程质量

- ESLint + Prettier（统一规范）
- Jest + Supertest（后端测试）
- React Testing Library（前端测试）

## 4. 单仓库协作策略

单仓库模式下，前后端与文档同仓管理，优势是：

- 需求、接口、前后端改动同一提交可追溯
- 教学展示时更易演示“需求-实现-测试-文档”闭环
- 可在同一迭代内快速联调并统一版本

建议的仓库内协作边界：

- `backend/`：API、鉴权、业务规则、数据访问
- `frontend/`：页面、组件、状态管理、交互
- `docs/`：接口文档、数据库设计、用户操作说明

## 5. 系统架构总览（高层）

### 5.1 核心流程

1. 用户注册/登录后获取 JWT  
2. 前端携带 JWT 调用后端受保护接口  
3. 后端完成权限校验与业务校验后访问 MySQL  
4. 活动审批与财务记录形成可追踪状态流并回显到前端  

### 5.2 模块划分

- 认证模块：注册、登录、鉴权中间件
- 社团模块：社团创建、成员加入/退出、成员角色管理
- 活动模块：活动创建、状态流转、审批意见
- 财务模块：收入/支出记录、分类统计、公示页面

## 6. 迭代计划（敏捷开发）

> 采用 3 个 Sprint（每个 2 周），每个 Sprint 均产出可演示增量版本。

### Sprint 1：基础能力与身份体系

目标：系统可登录、可鉴权、可访问受限页面

交付内容：

- 前后端依赖初始化与运行脚本
- 用户模型与认证接口（注册/登录）
- JWT 鉴权中间件
- 登录/注册页面与前端鉴权上下文
- 基础代码规范与目录约定

验收标准：

- 未登录用户无法访问受保护接口/页面
- 登录成功后可持续会话（token 持久化）

### Sprint 2：社团与活动审批主流程

目标：跑通“社团运营 + 活动审批”主链路

交付内容：

- 社团增删改查
- 成员加入、成员角色管理
- 活动创建、编辑、提交审批
- 系统管理员审批（通过/驳回）
- 活动状态可视化（draft/pending/approved/rejected）

验收标准：

- 社团管理员可提交活动
- 系统管理员可审批并留下意见
- 前端可按状态筛选活动

### Sprint 3：财务公示与整体打磨

目标：形成可用于课程展示的完整 MVP

交付内容：

- 财务收支录入与分类
- 财务公示页面（面向全体查看）
- 报表基础统计（月度/年度）
- 关键流程测试与缺陷修复
- API 文档与用户手册

验收标准：

- 社团财务记录可追踪、可展示
- 核心流程测试通过，系统可稳定演示

## 7. 数据库设计（MVP 最小闭环）

关键表：

- `users`：用户与系统角色
- `clubs`：社团信息
- `club_members`：社团成员关系与角色
- `activities`：活动主体与状态
- `approvals`：审批意见与审批结果
- `financial_records`：财务流水

设计要点：

- 所有业务表保留创建时间/更新时间
- 活动与审批拆分，避免状态历史丢失
- 通过外键保证社团、成员、活动、财务数据一致性

## 8. 关键工程规范（项目特定）

- API 返回结构统一（便于前端统一处理）
- 严格区分“认证失败（401）”与“权限不足（403）”
- 活动、财务相关列表必须支持分页
- 表单输入统一走前后端双重校验
- 仅对复杂业务逻辑添加注释，避免噪声注释

## 9. 风险与应对

### 风险 1：审批流逻辑膨胀

应对：先落地最小状态机（draft/pending/approved/rejected），不要在 MVP 阶段引入多级会签。

### 风险 2：权限边界混乱

应对：在后端中间件集中处理角色校验，前端只做展示层控制，不作为安全边界。

### 风险 3：联调效率低

应对：Sprint 1 末即冻结首版 API 合同，后续变更同步更新 `docs/API.md`。

## 10. 下一步落地清单（按执行顺序）

1. 在 `backend/`、`frontend/` 初始化 `package.json` 与基础脚本  
2. 建立后端 `app.js`、路由入口、统一错误处理中间件  
3. 建立前端路由骨架与认证上下文  
4. 配置 MySQL 连接与 Sequelize 模型/迁移  
5. 完成用户注册/登录接口与页面联调
